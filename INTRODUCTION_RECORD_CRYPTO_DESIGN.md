## 0. The construction in one paragraph

A `BlindedVouch` is a **dual credential**: a **classical BBS half** that carries *which-introducer* anonymity **unconditionally** (information-theoretic / everlasting — no adversary, quantum or otherwise, can ever strip it), and a **sovereign pure-Rust lattice half** that carries **post-quantum soundness** via a *threshold-established epoch credential that encodes no individual-introducer identity* (realization per §4.5). A verifier checks **both** (`AND`). The result: **anonymity is unconditional** (the BBS half is the sole, unbreakable carrier; the lattice half is built so even a total break of it reveals "an attested party" but never *which*), and **soundness is hedged asymmetrically across the quantum boundary** — classical era: holds if *either* half is unforgeable (BBS catches a classical-era bug in the young lattice port); but under the §2.3 PQ-capable adversary BBS is already forgeable, so **post-quantum the lattice half is the *sole* soundness root** (no BBS fallback for a lattice forgery bug — hence it is the most audit-critical component, HYP-330). This was **composition C2** of the three we evaluated — but the Codex DESIGN-review (P1) showed C2's shared-epoch-key realization has a soundness hole under a *compromised introducer*, which reopens the C2-vs-C3 choice (§4.4–§4.5). The dual-hybrid *shape* (BBS = unconditional anonymity, lattice = PQ soundness, AND-verify) stands; the lattice realization is now **DECIDED — C3** (Josh, 2026-06-13): individual threshold-*issued* credentials, blind-handed-off so D presents, bound to the BBS half — preserving one-introducer onboarding (core to the relationship model) with unconditional anonymity, at the cost of the novel cross-domain binding + a k-show nullifier + anonymous rate-limited issuance (the §4.4–§4.5 comparison is retained as decision rationale; the remaining §7 items + the Fable 5 + cryptographer + HYP-330 sign-off still gate any code).

**The reframe (the spine of the whole design).** The two halves are not "future-proof lattice + throwaway classical." They are **complementary opposites**:

| Property | Classical BBS | Lattice half |
|---|---|---|
| **Anonymity** (introducer hidden) | **everlasting / statistical** — unbreakable by *any* adversary, forever | computational (LWE/SIS) — strong but assumption-based |
| **Unforgeability** (no fake vouch) | classical (Gap q-SDH) — **quantum-broken** | **post-quantum** |

Read the diagonals: each half is strong on exactly the axis the other is weak. For a *privacy* system, BBS's **unconditional anonymity is the single strongest guarantee in the construction** — stronger than anything the lattice half offers — which is why we keep it and let it *own* anonymity, while the lattice half owns post-quantum soundness.

## 2. The contract the construction must satisfy

### 2.1 The trait boundary (HYP-337 shell — already merged)

```rust
trait VouchIssuer  { fn issue_vouch(&self, introduced: &DyadId, epoch: u64, anchor: &EpochAnchor)
                          -> Result<BlindedVouch, VouchError>; }
trait VouchVerifier{ fn verify_vouch(&self, vouch: &BlindedVouch, introduced: &DyadId, epoch: u64,
                          anchor: &EpochAnchor) -> Result<(), VouchError>; }

pub struct BlindedVouch(pub Vec<u8>);          // OPAQUE bytes — the construction owns the wire format
pub struct EpochAnchor { epoch: u64, bytes: Vec<u8> }   // derived from the Vita-Chain attested set
```

`BlindedVouch` being opaque bytes means the construction has **total wire-format freedom** (§3). The **verifier** side and the on-chain record / transport drop in with **zero rewiring** — `verify_vouch` already takes `(vouch, introduced, epoch, anchor)`, exactly what a verifier needs. **The issuer side, however, needs a shape change (Codex DESIGN-review):** the one-shot `issue_vouch(introduced, epoch, anchor) -> BlindedVouch` cannot express *interactive partially-blind* issuance — it hands the issuer the dyad and returns the finished vouch, which would let the issuer construct/link the token (defeating §4.1). The real flow is two messages: D builds a **blinded request** (commitment to the bound dyad + hidden per-record randomness) → B blind-signs → D unblinds + assembles. So the `VouchIssuer` trait must evolve to a request/response pair (e.g. `blind_request(introduced, epoch, anchor) -> (Request, UnblindState)` on D's side, `blind_sign(request, epoch, anchor) -> BlindResponse` on B's side, `unblind(response, UnblindState) -> BlindedVouch` on D's side). This is a **known shell extension to schedule with the real construction**, not zero-rewiring — flagged honestly here (the `StubVouchScheme`'s one-shot issue was a shell simplification).

**The *verifier* trait also needs record context for anti-replay (Codex DESIGN-review).** `verify_vouch(vouch, introduced, epoch, anchor)` authenticates none of `issued_at_ms` / `valid_through_ms` / a record nonce — so the per-record binding §4.2 requires of the lattice half cannot be *enforced*: a dyad with one valid vouch could copy the same `BlindedVouch` into another record for the same `(dyad, epoch)` with a fresh validity window + its own `chain_sig`. Fix (sign-off): either pass a **record-context** argument (the record nonce / validity window / a digest over the record's vouch-covered fields) into `verify_vouch` so the binding is checkable, **or** define a **ledger-wide nonce-uniqueness rule** (the chain rejects a second record reusing a vouch nonce). Until one is chosen, the anti-replay guarantee is stated but not enforceable through the current API.

### 2.2 The four properties (INTRODUCTION_RECORD §4, verbatim)

1. **Blindness (from the signer).** The introducer B does not learn the vouch token and cannot link its signing session to the published record. (B's own log can't reconstruct B's introductions.)
2. **Introducer anonymity (from the verifier).** V verifies against the *attested-introducer set / epoch anchor*, not B's individual key — V cannot tell *which* attested party vouched.
3. **Unlinkability (across records).** Two records (same or different B) are not linkable by their `blinded_proof`s.
4. **Soundness (no forgery).** D cannot produce a valid `BlindedVouch` without a real signature from *some* attested introducer — a Sybil cannot self-introduce.

### 2.3 The adversary (INTRODUCTION_RECORD §6)

A **global chain observer + a curious/compromised introducer + a colluding verifier set, all PQ-capable.** The construction must hold §4.1–§4.4 against all three. **"All PQ-capable" is why everlasting anonymity matters: a PQ verifier set still cannot break BBS's statistical anonymity.**

## 3. Verified crypto facts (the foundation — confidence-flagged)

These were verified against the primary literature this session (sources §8), upgrading the brief's hypotheses to facts:

- **[HIGH — verified] BBS is "proven unforgeable in the ROM under the Gap q-SDH assumption, and *statistically anonymous*."** Statistical anonymity = information-theoretic = holds against an unbounded/quantum adversary. The CFRG draft states the asymmetry explicitly (§6.9): unforgeability is computational and quantum-broken, while the proof's hiding properties remain quantum-resilient. *Mechanism:* a BBS proof re-randomizes the signature into a uniform group element and blinds every witness with uniform randomness in a Fiat–Shamir Schnorr SPK — perfectly/statistically hiding.
- **[HIGH — verified] Issuer-hiding (hiding *which* introducer among the attested set) is achievable with "everlasting issuer-hiding anonymity"** — Katz & Šefránek, eprint [2025/2080] (GGM; verification uses a verifier-held *policy key*).
- **[MEDIUM — verified-but-fresh] Public verification (no secret policy key) for issuer-hiding** is achieved by 2026 randomizable-key constructions — eprint [2026/369] ("via Randomizable Keys") and [2026/555] ("Improved Issuer Hiding," AGM, removes secret policy keys). ⚠️ These are 2026-fresh; whether a *single* construction gives **both** everlasting anonymity **and** public verification is the key BBS-side sign-off item (§7).
- **[HIGH — verified] Issuer-hiding presentations are compact — they do *not* scale with the number of issuers.** Constant-size regardless of attested-set growth.
- **[HIGH] Sizes.** BBS proof over BLS12-381 ≈ a few hundred bytes, sub-1 KB (G1 = 48 B compressed, scalar = 32 B; ~3 points + a handful of scalars for a 1-attribute vouch; issuer-hiding adds a constant). Lattice half ≈ tens of KB (brief §2–3). Combined record: tens of KB, single on-chain publish — within budget (INTRODUCTION_RECORD §5).

## 4. The construction

### 4.1 BBS half — the unconditional anonymity carrier

- **Scheme.** BBS (CFRG `draft-irtf-cfrg-bbs-signatures`) anonymous credential + an **issuer-hiding** presentation layer. Baseline: Katz–Šefránek everlasting issuer-hiding ([2025/2080]) with a **validator-held policy key** (fits the locked "validators verify" model). Upgrade-if-it-preserves-everlasting: the 2026 randomizable-key public-verification works ([2026/369], [2026/555]).
- **What "B vouches for D" is — blind issuance.** B issues D a one-attribute BBS credential **bound to the authenticated dyad** (binding **variant per the next bullet** — (a) issuer-visible bound message, or (b) committed dyad + ZK bond-equality), with a fresh **per-record blinding factor hidden from B** (cf. the spec's §4 `σ' = BlindSign_B(blind(T))` flow) so B's signing transcript `σ'` is **unlinkable** to the published `σ`/record. D presents an **issuer-hiding, unlinkable proof-of-possession** — the BBS component of `BlindedVouch` — proving "*an attested introducer in this epoch* vouched for D" and hiding which one **unconditionally**.
- **The dyad-binding is MANDATORY — a soundness requirement (Codex DESIGN-review P1).** A *fully*-hidden committed attribute would let a malicious requester get B to sign a commitment to a **different** dyad than the one B authenticated, then publish it for a Sybil — one legitimate vouch minting arbitrary introductions. So the credential MUST be bound to the authenticated dyad.
- **Two binding variants, with an HONEST blindness tradeoff (Codex DESIGN-review — §4.1 strict vs §6).** The partially-blind per-record factor hides the *signature randomness*, so a chain observer **without B's cooperation** cannot link signing sessions to records. But the two binding variants differ on what a *compromised B* learns:
  - **(a) Visible bound message** (dyad id is the issuer-visible partially-blind message). Simple, but since the same `introduced_dyad_id` is public in the record, a curious/compromised B can trivially match `(dyad, epoch)` in its own log and **reconstruct its own introductions**. This *contradicts the strict §4.1 reading* ("B's log cannot reconstruct B's introductions") — but is **consistent with §6** ("a colluding introducer learns nothing more than 'I vouched for D', which it already knows"). It does NOT leak *other* dyads' introducers or link across introducers. **Because §2.2 imports §4.1 verbatim, selecting (a) requires explicitly amending [INTRODUCTION_RECORD.md](INTRODUCTION_RECORD.md) §4.1 down to the §6 model (a parent-spec decision per the three-layer law, not just a construction choice)** — otherwise (a) fails the imported contract.
  - **(b) Committed dyad + ZK bond-equality** (B verifies, in zero-knowledge, that the committed attribute equals the dyad whose bond it authenticated, **without seeing the published dyad id** at issuance) — removes the *cryptographic-issuance* linkage. **But this alone does NOT guarantee strict §4.1 (Codex DESIGN-review):** B authenticates D's bond *out-of-band* (§4), so B already knows who D is; since `introduced_dyad_id` is *public in the record*, B can correlate "the dyad I met" ↔ the public record regardless of how blind issuance was. **Strict §4.1 therefore also requires the pairing/authentication flow to give B no handle linkable to the published `introduced_dyad_id`** (else even variant (b) achieves only §6). This makes strict §4.1 a *joint* property of the credential AND the pairing ceremony — arguably **§6 is the realistic model** for one-introducer onboarding. (Sign-off item.)
  - **This is a sign-off decision (§7):** accept (a)'s §6-bounded leakage for simplicity, or require (b) for strict §4.1. My v0.6 claim that the blinding factor alone prevents log-linkage was **wrong for variant (a)** and is corrected here.
- **What it provides:** §4.2 introducer-anonymity (unconditional, issuer-hiding), §4.3 unlinkability (statistical), and classical (q-SDH) individual unforgeability.
- **Crates.** Pairing: `bls12_381` (zkcrypto). BBS: an audited/maintained BBS impl — candidates `pairing_crypto` (DIF/MATTR), `docknetwork/crypto`. ⚠️ The **issuer-hiding randomizable-key layer is 2026 research → not in any audited crate** → we implement it on top of an audited BBS+`bls12_381` base, against the papers' test vectors. (Audit surface flagged for HYP-330.)

### 4.2 Lattice half — post-quantum soundness, no individual identity

*The lattice realization is the open §4.5 decision (C2-threshold vs C3); this section states what's common to both and where they differ — it does not mandate C2.*

- **The invariant both realizations MUST hold.** The lattice half carries **no recoverable individual-introducer identity** — so even a *total* break of its computational anonymity reveals only "an attested party authorized this," **never which party.** This is what lets BBS remain the sole (unconditional) anonymity carrier (§4.3). The two realizations achieve it differently: **C2-threshold** — a `t`-of-`n` threshold signature under a *shared* epoch key `K_epoch` (no per-introducer index in the signature); **C3** — an *individual* threshold-**issued** epoch credential whose anonymous proof-of-possession is **cryptographically bound to the BBS half** (the binding, not a shared key, is what prevents the lattice proof from independently leaking the introducer). **C3 ALSO requires *blind/anonymous issuance* of that epoch credential (Codex DESIGN-review):** if the threshold-issuing coalition can correlate the credential with B at issue time, a later lattice-proof break (exactly the scenario the hybrid hedges) that exposes the credential/witness would map records back to B *despite* BBS — so C3's unconditional-anonymity claim holds only if the credential is issued blindly (issuer learns nothing linkable to B). **This in turn requires an *anonymous-membership, rate-limited* issuance (Codex DESIGN-review):** the coalition must confirm the hidden requester is an attested member who has **not already drawn this epoch's credential** (a one-per-member-per-epoch issuance nullifier) — else a single member draws *multiple* unlinkable epoch credentials and bypasses the §4.5 per-credential rate-limit, breaking phantom-mint resistance. So C3 needs blind issuance **+** an issuance-time anonymous-membership rate-limit, both. The **eliminated** single-holder shared-`K_epoch` flow is neither (§4.2 compromise note + §4.5).
- **The lattice artifact MUST bind `introduced_dyad_id` + be anti-replay (Codex DESIGN-review).** Post-quantum the lattice half is the *sole* soundness root (BBS is forgeable), so binding it only to the epoch anchor is insufficient: a valid `P_lattice` from one record could be **replayed** with a forged BBS proof for a *different* dyad, defeating the sole-PQ-root guarantee. The signed/proven token MUST commit `introduced_dyad_id` **and** a per-record binding — an independent **nonce or presentation transcript** (NOT the `chain_sig` digest, which would be circular: `chain_sig` is computed over the assembled record that already contains this lattice artifact) — so each lattice artifact is valid for exactly one (dyad, record) and cannot be lifted. (Mirror of the §4.1 BBS dyad-binding — required on *both* halves.)
- **Scheme (standard-assumption base).** Beullens–Lyubashevsky–Nguyen–Seiler lattice blind signature, CCS 2023 ([2023/077]) — 20 KB, standard Module-SIS/LWE+NTRU, 2-round — as the base primitive in either realization. The token is **blind-signed / committed** so B cannot link its signing to the published record (§4.1) and the published artifact is re-randomized.
- **Threshold issuing power is MANDATORY in both realizations (Codex DESIGN-review P1).** The §6 adversary includes a *compromised introducer*, and **post-quantum the lattice half is the sole soundness mechanism** (BBS is forgeable), so a single party must never wield the epoch issuing power. C2-threshold: `K_epoch` is `t`-of-`n` threshold-**held** (no sub-`t` coalition can sign). C3: epoch credentials are `t`-of-`n` threshold-**issued** (no sub-`t` coalition can mint introducer credentials). Either way an explicit compromise threshold `t` is required — lattice threshold construction per Faller–Niot [2025/1566]. (My earlier "shared key + AND-verify saves it" reasoning was wrong: post-quantum the BBS AND-term provides no soundness.)
- **Residual compromise that threshold does NOT remove** (see §4.5): under one-introducer onboarding (C1/C3) a single compromised *member* can still vouch "as itself," boundable only by an added per-epoch **nullifier/rate-limit** (§4.5 iv) — *not* by revocation alone, since the introducer is unidentifiable. Only C2-threshold's per-vouch `t`-of-`n` removes it outright — at the cost of one-introducer onboarding.
- **Sovereign pure-Rust route.** LaZer (the reference lattice-ZK library) is **C / x86 / AVX-512 / Linux-amd64** — unusable on our ARM (Mac/iOS) targets and excluded by the sovereign-Rust lock. The path: **faithful transcription of the chosen scheme to pure Rust against published test vectors** (rule: never invented math), building where possible on the Rust lattice substrates `lattirust` / `Lazarus`. ⚠️ Test-vector availability + exact params at our security level are UNVERIFIED here and are sign-off + implementation items. This is **the highest-risk component in the arc** — a subtle sampling/rejection/constant-time bug = silent failure — which is why it gets the HYP-330 external audit.
- **What it provides:** §4.4 post-quantum soundness; it deliberately provides **no** individual-introducer anonymity (none is needed of it — BBS owns anonymity).

### 4.3 Composition + the property argument (realization-agnostic)

**Issue.** B (an attested introducer) gives D an individual **BBS** issuer-hiding credential on D. The **lattice** half is authorized by the attested set through a **threshold** operation whose exact form is the §4.5 decision — either *C2-threshold* (a `t`-of-`n` coalition co-signs D's lattice token per vouch) or *C3* (the set threshold-**issues** B an individual epoch credential at join; to vouch, B uses it to **blind-issue D a derived lattice token — a blind handoff — and D assembles + presents the unlinkable lattice proof**, cryptographically bound to the BBS half; B must NOT create the published artifact itself, or it could log + match it on-chain, breaking §4.1). **The eliminated single-introducer-shared-key flow — one B alone producing a shared-`K_epoch` signature — MUST NOT be implemented (§4.2, §4.5).**

**Present.** `BlindedVouch = encode(version ‖ P_bbs ‖ P_lattice)` — D's unlinkable BBS presentation plus the re-randomized lattice proof (realization per §4.5).

**Verify.** V checks **both** against the `EpochAnchor`: the BBS issuer-hiding presentation (against the epoch attested-key policy) **AND** the lattice proof (against the epoch lattice anchor — §5.2). **Both must pass.**

**Property argument:**

| Property | Argument | Holds against PQ adversary? |
|---|---|---|
| §4.1 Blindness | Both halves use blind issuance + D-side unlinkable presentation, so a chain observer can't link without B's cooperation. **Strict §4.1** (a compromised B can't reconstruct even its *own* introductions) needs binding **variant (b)** (committed dyad + ZK bond-equality) **AND** a pairing ceremony that hands B no handle linkable to the public `introduced_dyad_id` — *both*, since B authenticated D out-of-band. Variant (a), or (b) without that ceremony property, satisfies only the weaker **§6** model | strict §4.1: variant (b) **+** blind pairing; else §6 |
| §4.2 Introducer anonymity | **Unconditional** — BBS issuer-hiding is statistical; the lattice half carries **no recoverable individual identity** (per the §4.5 realization: C2-threshold's shared signature, or C3's BBS-bound proof), so it cannot leak which introducer even if fully broken | **yes — unconditionally** |
| §4.3 Unlinkability | BBS presentations statistically unlinkable; lattice signatures re-randomized per record; the only shared public values are `(D, epoch)`, already public in the record | yes |
| §4.4 Soundness | **AND-verify** → a forged vouch requires forging **both** the q-SDH BBS credential **and** the lattice artifact (the §4.5 realization's signature/credential-proof; for C3 also the cross-scheme binding). **Classical era:** holds if *either* is unforgeable (BBS catches a classical-era lattice-port bug). **Post-quantum:** BBS forgeable → soundness rests **solely on the lattice half** (a PQ lattice forgery bug is unhedged). | classical: if-either; **PQ: lattice-only** |

**The headline guarantees:** **anonymity is unconditional** (the strongest possible — and it survives a PQ verifier set, harvest-now-decrypt-later, and even a future break of the lattice half); **soundness is hedged asymmetrically** — *classical era:* if-either (BBS catches a classical-era lattice-port bug); *post-quantum:* lattice-only (BBS is forgeable, so a post-quantum lattice forgery bug is **not** hedged — the lattice half is the sole PQ soundness root and the most audit-critical component).

### 4.4 The realization alternatives — C1 / C2-threshold / C3 (the open comparison)

- **C1 (two per-introducer proofs, side-by-side) — dispreferred (gives up unconditional anonymity).** If the lattice half is a per-introducer membership proof, its hidden witness *is* "which introducer," and a classical cryptanalysis break or **a bug in our own lattice port** would deanonymize the graph *despite* BBS (side-by-side, record anonymity is capped at the **weaker** half). That is the exact "silent graph deanonymization" failure the issue names — unhedged for anonymity. C2 fixes this by giving the lattice half no individual identity to leak.
- **C3 (cross-scheme binding — split the introducer witness across both halves) — the SELECTED realization (see §4.5/§4.6; this comparison is retained as rationale).** *On the anonymity axis alone*, C3 buys nothing over C2: BBS anonymity is unconditional, so "break both" is unachievable and C3's anonymity collapses to the same unconditional ceiling C2 reaches — for the cost of a novel cross-domain ZK proof (proving one hidden value across the elliptic-curve and module-lattice worlds; no off-the-shelf pattern; highest audit risk). **That was the basis for preferring C2 — but it only considered anonymity.** Codex's P1 shows the *soundness-under-compromise* axis distinguishes them — but **no one-introducer scheme is fully per-vouch compromise-safe** (see §4.5): C2's no-individual-identity property forces a *shared/threshold* lattice key, whose single-introducer realization (shared key) has the catastrophic compromise hole (one leak mints *unattributable* vouches), while only its *threshold-per-vouch* realization (t-of-n co-sign per vouch, onboarding friction) makes a single compromise insufficient to vouch at all. C3 uses **individual, threshold-*issued*** credentials — one introducer vouches alone, threshold *issuance* blocks *phantom* (outsider) credentials, *and* the binding keeps anonymity unconditional — **but a compromised member can still reuse its own credential to vouch for Sybils "as itself"** (boundable only by an added per-epoch nullifier/rate-limit, §4.5 iv — *not* by revocation, since verifiers don't identify which introducer). So the honest distinction is: C2-threshold buys *per-vouch* compromise resistance at the cost of one-introducer onboarding; C3 keeps one-introducer onboarding + unconditional anonymity but needs the nullifier for compromised-member accountability. **The C2-vs-C3 choice is DECIDED — C3** (§4.5); C2-threshold is the documented fallback only.

### 4.5 The per-vouch-authorization trilemma — DECIDED: C3 (rationale retained)

*This section's comparison is **historical decision rationale**, not an open question — the construction is DECIDED (C3, §0/§1). C2-threshold remains documented only as the explicit fallback if C3 fails the cryptographer sign-off (§4.6).*

Four properties we want — and **the second compromise property (iv) is fundamentally incompatible with one-introducer onboarding (i)**, so no scheme holds all four:

- **(i) one-introducer onboarding** — a new dyad is introduced by *one* attested party it actually knows (the intimate pairing-ceremony model; low join friction).
- **(ii) unconditional anonymity** — the lattice half carries no recoverable individual identity, so record anonymity is the BBS half's unconditional level; otherwise a break of the lattice half's computational anonymity deanonymizes the graph despite BBS.
- **(iii) phantom/outsider-mint resistance** — an *outsider* (non-member), or a leak short of the issuing threshold, cannot fabricate introducer credentials or mint *unattributable* vouches. (This is the property C2-shared catastrophically fails: post-quantum, one leaked shared key mints unlimited vouches attributable to no one.)
- **(iv) per-vouch compromised-*member* resistance** — a single *compromised attested member* cannot mint Sybil vouches *at all*. This requires `t`-of-`n` participation **per vouch**, which **inherently breaks (i)**. Under one-introducer onboarding a compromised member can always vouch "as itself" (verifiers require only "*some* attested introducer" and don't identify which), so (iv) cannot be cryptographically prevented. **And "revocation-bounded" is itself not free (Codex DESIGN-review):** because verifiers cannot identify *which* introducer vouched, there is no evidence path to attribute bad vouches or pick a credential to revoke — so bounding the abuse needs an **added accountability mechanism**. The privacy-preserving answer is a **k-times-anonymous / one-show-token** rate-limit (caps how many vouches one credential can issue per epoch *without* deanonymizing). **Caveat (Codex DESIGN-review):** a *naive* per-epoch nullifier that permits >1 vouch becomes a **stable public tag** — every record sharing it links to the same hidden introducer, violating §4.3 unlinkability. So the mechanism MUST be a proper k-show scheme (fresh unlinkable per-use slots, k-bounded), *or* the cap is exactly **one vouch per credential per epoch** (a one-show nullifier — unlinkable, at a throughput cost). An **opening authority** would also bound abuse but **breaks unconditional anonymity** (disfavored). Absent any of these, compromised-member abuse is *unbounded until out-of-band detection*. (§7 sign-off item — the k-show design + throughput tradeoff.)

| | Lattice half | (i) one-introducer | (ii) uncond. anon | (iii) phantom-mint resist | (iv) per-vouch member-compromise | Cost |

| **C1** | per-introducer key + ZK membership | ✅ | ❌ (computational — break deanonymizes) | ✅ | ⚠️ needs per-epoch nullifier (iv) | lowest crypto risk; weakest anonymity |

### 4.6 C3 cross-domain binding — design + feasibility (the make-or-break)

C3's one hard part is binding the BBS (elliptic-curve / BLS12-381) presentation to the lattice (module-lattice) presentation so the introducer witness is recoverable only from **both** — *anonymity-if-either*, with the unconditional side dominating. This section sketches a buildable construction grounded in existing literature + an honest feasibility read. **It is a proposal for the cryptographer sign-off, not a finished scheme.**

**Why a mere shared transcript is insufficient.** Binding the two halves by a common Fiat–Shamir challenge over the record (the "easy" binding) gives *soundness* binding (can't lift one half to another record) but **not** anonymity-if-either: proving "an *attested member* authorized this" anonymously inherently carries "*which* member" as a hidden witness, so if the lattice proof's (computational) ZK breaks, the introducer leaks regardless of a shared transcript. C3 needs the witness itself **split** across the domains.

**The construction sketch (additive-split commit-and-prove).**
1. Let `w` be the introducer's secret identifier. Split it additively: `w = w_p + w_l` with `w_p` random.
2. Commit `w_p` in a **Pedersen commitment over BLS12-381** — *perfectly* hiding (information-theoretic). The BBS issuer-hiding presentation proves `w_p`'s commitment corresponds to an attested introducer's BBS credential (unconditional anonymity).
3. Commit `w_l` on the **lattice side**; the lattice credential proof proves `w_l`'s commitment corresponds to a valid epoch credential (post-quantum unforgeability).
4. A **cross-domain commit-and-prove link** proves `w_p + w_l` equals the bound attested identity (and binds the record nonce + `introduced_dyad_id`, §4.2), using the **Pedersen↔lattice same-value technique** (Asiacrypt 2014, "Better ZK Proofs for Lattice Encryption…" — prove a Pedersen commitment and a lattice commitment hold the same value by matching coefficients *over the integers*, no modular reduction on the committed terms; this exact bridge was used to build a *hybrid group signature*). The Pedersen side of the link is a **Schnorr proof → perfect ZK**, so `w_p` is revealed to *no one*, even unbounded.

**Why this gives anonymity-if-either.** `w_p` is perfectly hidden (Pedersen + perfect-ZK link). A break of the lattice side (its computational ZK / SIS-LWE hiding) reveals at most `w_l`; recovering `w = w_p + w_l` still needs `w_p`, which is information-theoretically hidden — so the introducer stays hidden **unconditionally**, regardless of the lattice side. Soundness: a forgery needs *both* a valid BBS credential on `w_p` (q-SDH, classical) *and* a valid lattice credential on `w_l` (PQ) satisfying the link → **classical era:** holds if *either* is unforgeable; **post-quantum:** BBS is forgeable, so soundness rests **solely on the lattice half** (the asymmetry of §0/§4.3 — a PQ lattice forgery bug is unhedged).

**Feasibility read (confidence-flagged).**
- **[MEDIUM — search-grounded, primary PDF not fetched] The core primitive exists.** Hybrid commit-and-prove ZK spanning Pedersen + lattice commitments is published, and the Asiacrypt-2014 bridge proves same-value across the two domains and *demonstrably* powered a hybrid (lattice-privacy + DL-security) group signature. So "bind a value across EC and lattice in ZK" is **not** unprecedented.
- **[The genuinely novel part] We need the *axes flipped* vs the precedent.** Asiacrypt-2014's hybrid puts *anonymity on the lattice side*; we want anonymity on the *unconditional Pedersen/BBS side* + soundness if-either. The additive-split + perfect-ZK-Pedersen-link above is the proposed flip — plausible, but it is the thing a cryptographer must validate (does the composed link leak `w_p` through the lattice side or the relation? is the Pedersen part genuinely perfect-ZK in-composition?).
- **[UNVERIFIED] Sizes/params** (the keystone PDF 403'd) — and the **sovereign pure-Rust** port of the bridge + the lattice ZK is the highest-risk build in the arc (HYP-330).

**Cryptographer sign-off targets for C3 (this section):** (1) validate the additive-split flip gives unconditional anonymity in the *full* composition; (2) the concrete Pedersen↔lattice link scheme + params at our security level + current cryptanalysis; (3) sizes; (4) the sovereign-Rust constant-time port plan + test vectors. If (1) cannot be established soundly, C3 reopens (fallback: C2-threshold, accepting onboarding friction).

## 5. Wire format + EpochAnchor

### 5.1 `BlindedVouch` bytes

```
BlindedVouch = VOUCH_VERSION (u16)
             ‖ len(P_bbs)     (u32) ‖ P_bbs
             ‖ len(σ_lattice) (u32) ‖ σ_lattice
```

- Length-prefixed, version-tagged, canonical (a frame-cap precedes decode, per the codec-safety rule used in `introduction_record_sync`). Total ≈ tens of KB.
- `VOUCH_VERSION` lets the BBS-only interim (if the build is ever re-phased) and the full dual record be distinguished — though the locked build order is both-at-once, so v1 = dual.

### 5.2 `EpochAnchor` bytes — derivable from the Vita-Chain attested set, **no extra trusted setup**

**Self-delimiting, versioned, tagged encoding (Codex DESIGN-review P2)** — a raw concatenation is not unambiguously parseable (the BBS policy may be a full key-set *or* a root; revocation is optional), so the anchor is framed like `BlindedVouch`: a version, then a tag byte + length prefix per field. Verifiers reject unknown tags / trailing bytes.

The tag set for the lattice field is **conditional on the §4.5 realization** (the anchor does not pre-freeze a C2-only format): C2-threshold publishes the shared epoch verification key (`tag=LATTICE_KEPOCH_PUBLIC`); C3 publishes the epoch issuer key / accumulator for the individual threshold-issued credentials (`tag=LATTICE_EPOCH_ISSUER`). A verifier selects its lattice check by the present tag.

```
EpochAnchor.bytes = ANCHOR_VERSION (u16)
                  ‖ field{ tag=BBS_POLICY_KEYSET | BBS_POLICY_ROOT,            len(u32), bytes }
                  ‖ field{ tag=LATTICE_KEPOCH_PUBLIC | LATTICE_EPOCH_ISSUER,   len(u32), bytes }  // realization per §4.5
                  ‖ field{ tag=REVOCATION_ROOT (OPTIONAL),                     len(u32), bytes }   // absent ⇒ field omitted, not zero-length-ambiguous
```

- **BBS policy** (`tag` distinguishes keyset vs root) = the attested introducer BBS public keys for the epoch, or a Merkle/accumulator root over them; with randomizable-key public verification this is published and verification is public. Derived directly from the chain's attested set → no trusted setup.
- **Lattice anchor** (`tag` distinguishes the §4.5 realization) = C2-threshold's shared epoch verification key, **or** C3's epoch issuer key/accumulator — both **threshold-established** by the attested set at epoch start via DKG (§4.2). Derived from the attested set → no trusted setup. The single-holder shared key is eliminated regardless of tag.
- **Revocation** (⚠️ sign-off): present (a lattice accumulator root, Libert et al. / LaZer-style, over non-revoked introducers) or omitted (epoch-rotation-only) — the tag's presence/absence is unambiguous in the framing above.

### 5.3 Epoch model + churn (⚠️ sign-off — INTRODUCTION_RECORD brief Q5)

- `epoch` scopes both the BBS policy and the lattice epoch key/credentials (§4.5). Per-epoch rotation gives the **forward-secret graph** property (§6: a leaked key ≠ retroactive graph) and is the anonymity-set boundary ("attested introducers in this epoch").
- Epoch length trades anonymity-set size (longer = bigger set) against re-keying cost (threshold DKG for the lattice epoch key/issuance + BBS policy refresh each epoch) and record-refresh frequency (`valid_through_ms`). **Epoch length is a sign-off parameter.**

### 5.4 Public vs validator verification

Verification can be **public** (randomizable-key issuer-hiding, no secret policy key) — so light clients *may* verify, not only validators. The locked "validators verify" stands as the baseline (and is required for the everlasting [2025/2080] policy-key variant if the public-verification variant cannot also be everlasting); public verification is a strict upgrade where available.
